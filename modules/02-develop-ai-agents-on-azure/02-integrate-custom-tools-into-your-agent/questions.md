**Q27.** In the `FunctionTool` parameters JSON schema example, what does setting `"additionalProperties": False` combined with `"required": ["location"]` enforce?

**Q58.** A retail company wants an agent to (1) check current stock and reorder from suppliers automatically via their existing inventory system's REST API, described by a published OpenAPI 3.0 spec with no authentication required, and (2) run a small in-app Python function that calculates a loyalty discount using business logic that lives only in the agent's own codebase. Which two tool types should be registered, and which auth-detail class fits the first requirement?
A. OpenAPI specification tool with `OpenApiAnonymousAuthDetails`, plus function calling for the discount logic
B. Azure Functions with `OpenApiApiKeyAuthDetails`, plus Azure Logic Apps for the discount logic
C. Function calling for both requirements, since custom logic always means function calling
D. OpenAPI specification tool with `OpenApiManagedIdentityAuthDetails`, plus Azure Functions for the discount logic
**Answer:** A — The already-described external API with no auth maps to an OpenAPI specification tool using `OpenApiAnonymousAuthDetails` (one of the three supported auth types: anonymous, API key, managed identity). The in-app Python function with logic living only in the agent's own code maps to function calling (via `FunctionTool`), not Azure Functions (which is for externally deployed serverless compute reached via storage queue).

**Q59.** Your team already has a deployed Azure Function that reacts to a storage queue message and returns weather data on an output queue. You register it as a tool on a Foundry agent using `AzureFunctionTool`. A teammate asks: "Do we also need to write code in our agent application that calls this Azure Function whenever a user asks about weather?" What's the correct answer, and why?
A. Yes — the developer must explicitly invoke the Azure Function tool in application code for every single relevant user message received
B. No — tool use is declarative; once registered with a meaningful name and well-documented parameters, the agent itself decides at runtime whether and how to invoke the Azure Function
C. Yes, but only if `strict=True` is explicitly set on the `AzureFunctionTool` definition object itself
D. No — but only because Azure Functions tools are treated as the sole exception to the general declarative tool-use pattern
**Answer:** B — The module's key conceptual callout states tool use is declarative across all three custom tool types (function calling, Azure Functions, OpenAPI): the agent itself decides when to call a registered tool based on the conversation, without the developer writing explicit invocation logic.

**Q60.** A developer is choosing between Azure Functions and Azure Logic Apps for a new agent capability that needs custom C#/Python code to run event-driven, serverless compute with sub-second latency triggered by an HTTP request, versus a scenario that just needs to visually wire together several existing SaaS connectors with minimal code. Which option fits each scenario, and what's the key distinguishing factor?
A. Azure Functions fits the serverless custom-code/HTTP-trigger scenario; Azure Logic Apps fits the low-code/no-code connector-wiring scenario — the distinguishing factor is code-based serverless compute (triggers/bindings) vs. a low-code/no-code workflow connector
B. Both scenarios require Azure Logic Apps, since Azure Functions cannot use HTTP triggers or bindings for event-driven, sub-second-latency serverless compute at all, per the module's own description
C. Both scenarios require Azure Functions, since Azure Logic Apps cannot call or wire together existing SaaS connectors without writing custom code, contradicting its low-code design entirely
D. Azure Logic Apps fits the first scenario because it supports triggers and bindings for sub-second HTTP-triggered compute, not Azure Functions, which the module ties only to low-code connector wiring between apps and services
**Answer:** A — Azure Functions provides serverless, event-driven compute with triggers (e.g., HTTP) and bindings for custom code; Azure Logic Apps is explicitly described as the low-code/no-code option for connecting apps, data, and services. Triggers/bindings are an Azure Functions concept, not a Logic Apps one, per the module.
